What it is

The foundational text task, done properly

Document classification sounds mundane until you build it: take free-form text of any length and assign it a label — spam or not, news topic, ticket department, sentiment. Almost every practical NLP system has this shape somewhere inside it, which is exactly why it's worth rebuilding carefully rather than importing.

This build walks the full arc, deliberately. It starts from the classic, interpretable approach — turn text into TF-IDF vectors and train a linear classifier — and then contrasts it with dense embeddings, so you feel the trade-off between a transparent model you can read and a stronger one you mostly can't.

Architecture

How a document is classified

Every document follows the same represent-then-decide pipeline:

  1. Preprocess

    Tokenise and clean the raw text into a consistent form.

  2. Vectorise

    Turn the document into a fixed-length vector — TF-IDF or embedding.

  3. Train

    Fit a classifier on the labelled training vectors.

  4. Predict

    Assign a category to each unseen document.

  5. Evaluate

    Score per-class performance and read the confusion matrix for blind spots.

Reflection

What rebuilding it taught me

  • TF-IDF is hard to beat. On many real tasks a simple, readable linear model on TF-IDF is shockingly close to the fancy alternative — and you can explain it.
  • Preprocessing is where accuracy hides. Tokenisation and stopword choices move the score more than the model swap often does.
  • Interpretability is a feature. Being able to read which words drove a decision is worth real accuracy in anything that touches people.
  • Accuracy is a trap on imbalance. A classifier can score 95% and be useless; per-class F1 keeps you honest.
  • Master this, generalise everywhere. Spam, topics, sentiment, routing — once the pipeline is yours, the task just changes labels.
